## Role/Persona (角色/人设)

### 作用
角色设定通过指定AI应该扮演的身份，聚焦响应的视角、词汇、专业深度和沟通风格。

### 为什么重要
- **激活相关知识域**：不同角色激活AI知识库的不同部分
- **统一回答风格**：确保整个回答的一致性
- **调整专业深度**：控制技术细节的深度和广度
- **匹配受众水平**：让AI使用适当的语言复杂度

### 构建角色提示的模式

#### 专家模式
```
你是一位[领域]专家，具有[X]年经验，专长于[具体专业领域]。

示例：
你是一位机器学习专家，具有10年经验，专长于深度学习和自然语言处理。
```

#### 专业人士模式
```
你是一位[职业]，在[行业/组织类型]工作，专注于[工作内容]。

示例：
你是一位产品经理，在B2B SaaS公司工作，专注于企业级应用的用户体验设计。
```

#### 教师模式
```
你是一位[学科]老师，擅长为[目标受众]解释复杂概念，教学风格是[风格特征]。

示例：
你是一位数学老师，擅长为初中生解释抽象代数概念，教学风格是耐心且多用比喻。
```

#### 复合角色模式
```
你是一位[角色A]，同时也是[角色B]。你结合了[A的优势]和[B的优势]。

示例：
你是一位心理学家，同时也是一位资深程序员。你结合了心理学的洞察力和工程学的精确性。
```

### 常见错误

**错误1：角色过于模糊**
- ❌ "你是一个专家"
- ✅ "你是一位具有8年经验的Python后端开发工程师，专长于高并发系统设计"

**错误2：角色与任务不匹配**
- ❌ 角色是"诗人"，任务是"审查数据库设计"
- ✅ 角色是"数据库架构师"，任务是"审查数据库设计"

**错误3：添加无关的角色特征**
- ❌ "你是一位AI助教，喜欢摇滚音乐，每周去健身房三次"
- ✅ "你是一位AI助教，善于解释复杂概念，耐心且鼓励学生"

### 最佳实践
1. **具体明确**：使用具体的职业名称和专业领域
2. **经验量化**：添加经验年限增加可信度
3. **专业聚焦**：指定专长领域，避免泛泛而谈
4. **风格特征**：添加沟通风格特征（耐心、严谨、幽默等）
5. **场景关联**：让角色与实际使用场景相关联

## Context/Background (上下文/背景)

### 作用
提供AI完成任务所需的所有背景信息，包括项目详情、当前状态、目标和约束。

### 为什么重要
- **AI没有记忆**：每次对话都是全新的开始
- **消除歧义**：背景信息帮助AI准确理解需求
- **提高相关性**：确保回答与具体情境相关
- **避免假设**：防止AI做出错误的假设

### 上下文检查清单

在提供上下文前，检查以下问题：
- [ ] AI知道我在做什么项目吗？
- [ ] AI了解我的技术栈或工具吗？
- [ ] AI知道我的目标和期望吗？
- [ ] AI了解当前的约束条件吗？
- [ ] AI知道目标受众是谁吗？
- [ ] AI了解当前的状态和进度吗？

### 上下文的组成部分

#### 项目信息
```
项目类型：[类型]
技术栈：[技术列表]
团队规模：[人数]
当前阶段：[阶段]
```

**示例**：
```
项目信息：
- 项目类型：电商网站后台管理系统
- 技术栈：React前端 + Node.js + PostgreSQL数据库
- 团队规模：5名开发人员
- 当前阶段：MVP开发阶段，预计2个月后上线
```

#### 当前状态
```
已完成：[已完成的内容]
当前挑战：[面临的问题]
目标：[想要达成的结果]
```

**示例**：
```
当前状态：
- 已完成：用户认证模块、商品展示模块、购物车功能
- 当前挑战：订单处理流程存在性能瓶颈，高并发时响应时间超过5秒
- 目标：将订单处理时间优化到1秒以内
```

#### 约束条件
```
技术约束：[技术限制]
时间约束：[时间限制]
资源约束：[资源限制]
业务约束：[业务规则]
```

**示例**：
```
约束条件：
- 技术约束：必须兼容IE11浏览器
- 时间约束：需要在2周内完成优化
- 资源约束：只能使用现有的数据库架构，不能添加新表
- 业务约束：不能改变现有的订单流程逻辑
```

#### 受众信息
```
目标受众：[受众类型]
知识水平：[专业程度]
使用场景：[使用环境]
```

**示例**：
```
受众信息：
- 目标受众：初级开发人员，刚入职1-3个月
- 知识水平：了解基本的编程概念，但不熟悉项目架构
- 使用场景：入职培训期间学习项目代码
```

### 常见错误

**错误1：信息不足**
- ❌ "帮我优化这段代码"
- ✅ "这是一个处理用户上传图片的Python函数，需要优化内存使用，因为服务器只有4GB RAM"

**错误2：信息无关**
- ❌ "我今年30岁，有两个孩子，住在北京"（与代码优化无关）
- ✅ "这是一个教育类应用，需要考虑家长对隐私的关注"

**错误3：假设AI知道**
- ❌ "像之前那样做"（AI没有之前对话的记忆）
- ✅ "参考标准的REST API设计模式"

### 最佳实践
1. **提供充足但不冗余**：足够的信息确保理解，避免无关细节
2. **结构化呈现**：使用列表或分段使信息清晰
3. **明确边界**：说明什么可以做什么不能做
4. **具体化指标**：提供具体的数字和指标而非模糊描述
5. **状态清晰**：明确当前处于什么阶段

## Task/Instruction (任务/指令)

### 作用
明确告诉AI要执行什么具体操作，这是提示词的核心。

### 为什么重要
- **直接决定输出**：任务描述的质量直接影响结果质量
- **避免误解**：清晰的指令防止AI做错事情
- **设定期望**：明确知道应该得到什么
- **可衡量性**：明确的任务便于评估结果

### 任务设计原则

#### 使用具体动词
避免模糊动词，使用精确的行动动词：

| 模糊动词 | 精确动词 |
|---------|---------|
| 帮助我 | 分析/审查/优化/设计 |
| 处理 | 转换/提取/格式化/过滤 |
| 写 | 撰写/创作/总结/扩展 |
| 看 | 检查/评估/比较/诊断 |

#### 明确任务范围
说明任务的边界和范围：
- 要做什么
- 不要做什么
- 重点关注什么

#### 指定深度和广度
- **深度**：需要多详细？概览/详细/深度分析
- **广度**：涵盖哪些方面？只核心/全面/特定角度

### 任务模式

#### 创建类任务
```
创建/撰写/生成/设计/开发 [目标对象]

示例：
- 撰写一份500字的用户手册
- 设计一个登录页面的用户流程
- 生成10个产品创意
```

#### 分析类任务
```
分析/评估/审查/诊断/比较 [对象] 从 [角度]

示例：
- 分析用户反馈中的情感倾向
- 评估代码的安全性风险
- 比较两种技术方案的优劣
```

#### 转换类任务
```
转换/翻译/格式化/重构 [源对象] 为 [目标格式]

示例：
- 将这段文本转换为JSON格式
- 翻译为西班牙语，保持技术准确性
- 重构为函数式编程风格
```

#### 问题解决类任务
```
解决/修复/调试/优化 [问题]

示例：
- 解决这段代码中的内存泄漏问题
- 优化数据库查询性能
- 调试登录失败的原因
```

### 任务优化示例

**原任务**：
```
帮我看看这个提示词
```

**优化版本1**（明确任务类型）：
```
审查这个提示词，识别可以改进的地方
```

**优化版本2**（添加关注点）：
```
审查这个提示词，重点关注：1）是否缺少关键组件 2）是否有歧义表述 3）输出格式是否明确
```

**优化版本3**（添加深度和范围）：
```
深入审查这个提示词，识别所有可以改进的方面。分析其清晰度、完整性、有效性，并按优先级列出改进建议。
```

### 常见错误

**错误1：任务模糊**
- ❌ "帮我处理这个"
- ✅ "将这段文本中的所有邮箱地址提取为列表"

**错误2：任务范围过大**
- ❌ "帮我写一个完整的电商网站"
- ✅ "设计一个电商网站的购物车功能，包括添加商品、修改数量、计算总价"

**错误3：缺少具体标准**
- ❌ "让它好一点"
- ✅ "提升响应速度至少50%"

### 最佳实践
1. **单一明确**：每个提示词聚焦一个主要任务
2. **可操作**：使用具体的行动动词
3. **可衡量**：设定明确的标准和指标
4. **分层次**：复杂任务分解为子任务
5. **有边界**：明确任务的范围和限制

## Constraints/Rules (约束/规则)

### 作用
设定输出的限制条件和规则，防止常见问题并确保相关性。

### 为什么重要
- **控制输出质量**：避免过长、过短或跑题
- **确保一致性**：统一风格和格式
- **防止问题**：避免常见错误和不当内容
- **满足特定需求**：符合业务或技术要求

### 约束类型

#### 长度约束
```
保持回复在[X字/X段落]以内
提供恰好[X]个建议
每条建议不超过[X字]
```

**示例**：
```
- 将回复控制在300字以内
- 提供恰好5个优化建议
- 每个说明不超过2句话
```

#### 内容约束
```
不要包含[内容类型]
只关注[特定方面]
避免[不当内容]
```

**示例**：
```
- 不要包含代码示例
- 只关注性能优化，不考虑代码可读性
- 避免使用过于专业的术语
```

#### 风格约束
```
使用[正式/非正式/学术/幽默]的语气
写成[第一人称/第三人称]
使用[简单/专业]的语言
```

**示例**：
```
- 使用友好的、鼓励的语气
- 写成给初学者的教程风格
- 使用简洁明了的语言，避免过度复杂的句子
```

#### 格式约束
```
使用[特定格式]
遵循[风格指南]
不使用[特定元素]
```

**示例**：
```
- 使用Markdown格式
- 遵循PEP 8代码规范
- 不要使用emoji表情符号
```

#### 范围约束
```
只考虑[条件]
限制在[范围]内
排除[内容]
```

**示例**：
```
- 只考虑Python 3.8+的语法特性
- 限制在免费工具范围内
- 排除需要额外依赖的方案
```

#### 逻辑约束
```
按[顺序/结构]组织
确保[逻辑要求]
避免[逻辑错误]
```

**示例**：
```
- 按重要性从高到低排序
- 确保每个建议都有具体的实施步骤
- 避免相互矛盾的建议
```

### 约束组合示例

```markdown
约束条件：
- 长度：总回复不超过500字
- 内容：只关注用户体验方面，不考虑技术实现
- 风格：使用简洁、直接的语言
- 格式：使用无序列表
- 范围：只适用于移动端场景
- 逻辑：建议按实施难度从易到难排序
```

### 常见错误

**错误1：约束过多**
- ❌ 10+条约束，AI难以全部满足
- ✅ 3-5条最重要的约束

**错误2：约束相互冲突**
- ❌ "保持简洁但也要详细"
- ✅ "简洁但完整"

**错误3：约束不合理**
- ❌ "分析整本书，但回复不超过100字"
- ✅ "提供整本书的100字摘要"

### 最佳实践
1. **优先级明确**：最重要的约束放在前面
2. **数量适中**：3-5条关键约束即可
3. **相互协调**：确保约束之间不冲突
4. **可验证**：约束应该能够被检查和验证
5. **灵活性**：为创造性任务留出空间

## Output Format (输出格式)

### 作用
指定AI应该以什么格式、结构呈现输出结果。

### 为什么重要
- **提高可用性**：结构化输出更易使用
- **便于处理**：特定格式便于程序处理
- **确保完整性**：结构化确保所有必需内容都包含
- **便于比较**：统一格式便于对比多个结果

### 常见输出格式

#### 列表格式

**无序列表**：
```
使用无序列表，每项简短明确
```

**有序列表**：
```
使用有序列表，每项包含[内容要求]
```

**嵌套列表**：
```
- 一级标题
  - 二级项目
    - 三级细节
```

#### 结构化文本

**分段结构**：
```
## 标题
[段落内容]

### 小标题
[段落内容]
```

**段落模板**：
```
结构你的回复为：
1. 概述（2-3句话）
2. 关键发现（3-5个要点）
3. 详细分析（每个要点2-3句话）
4. 结论与建议（1-2句话）
```

#### 表格格式

**Markdown表格**：
```
| 列1 | 列2 | 列3 |
|-----|-----|-----|
| 内容 | 内容 | 内容 |
```

**表格要求**：
```
以表格形式返回，包含以下列：
- 列名1：说明
- 列名2：说明
- 列名3：说明
```

#### 数据格式

**JSON格式**：
```
返回有效的JSON，不包含其他文本：
{
  "key1": "value1",
  "key2": "value2",
  "key3": ["item1", "item2"]
}
```

**YAML格式**：
```
key1: value1
key2:
  - item1
  - item2
```

#### 代码格式

```
使用```代码块```包裹所有代码
包含必要的注释
标明代码语言（python, javascript等）
```

### 格式指定技巧

#### 明确结构模板
```
请按以下结构组织回复：

## [标题1]
[内容]

## [标题2]
- [要点1]
- [要点2]

## [标题3]
[内容]
```

#### 指定每部分内容
```
为每个改进建议提供：
1. 问题描述（1句话）
2. 影响分析（2-3句话）
3. 具体方案（步骤列表）
4. 预期效果（1句话）
```

#### 添加格式要求
```
- 使用Markdown格式
- 关键术语使用**粗体**
- 代码使用`行内代码`或代码块
- 段落之间空一行
```

### 格式示例

**简单列表格式**：
```
以无序列表形式返回5个最佳实践，每条不超过20字。
```

**结构化报告格式**：
```
结构你的分析报告如下：

# 标题

## 执行摘要
[2-3句话的总体概述]

## 问题分析
### 问题1：[名称]
- **影响**：[1句话]
- **根因**：[2-3句话]
- **证据**：[具体数据或观察]

### 问题2：[名称]
[同上结构]

## 解决方案
### 方案1：[名称]
- **描述**：[2-3句话]
- **实施**：[步骤列表]
- **预期**：[1句话]

## 结论
[最终建议，1-2句话]
```

**JSON数据格式**：
```
返回纯JSON格式（无其他文本），结构如下：
{
  "analysis": {
    "status": "success|error",
    "timestamp": "ISO时间戳",
    "summary": "一句话摘要"
  },
  "findings": [
    {
      "id": "唯一标识",
      "category": "类别",
      "description": "描述",
      "severity": "高|中|低",
      "recommendation": "建议"
    }
  ],
  "metadata": {
    "version": "1.0",
    "source": "数据来源"
  }
}
```

### 常见错误

**错误1：格式未定义**
- ❌ "给我结果"
- ✅ "以表格形式返回，包含列：问题、严重程度、建议"

**错误2：格式过于复杂**
- ❌ 要求复杂的嵌套结构，AI难以准确遵循
- ✅ 使用清晰、简单但完整的结构

**错误3：格式与任务不匹配**
- ❌ 创意写作任务要求严格的JSON格式
- ✅ 给予一定的格式灵活性

### 最佳实践
1. **提供模板**：给出明确的结构模板
2. **说明每部分**：告诉每部分应该包含什么
3. **保持简单**：优先简单清晰的结构
4. **示例展示**：如果可能，展示期望的格式示例
5. **权衡灵活性**：根据任务调整格式严格程度

## Examples (示例)

### 作用
通过提供输入-输出示例，向AI展示期望的行为模式和输出质量。

### 为什么重要
- **最强大的指导**：示例比任何描述都有效
- **精确传达期望**：展示比解释更准确
- **减少歧义**：示例消除了理解偏差
- **提升一致性**：确保输出与示例一致

### 示例类型

#### 输入-输出示例
```
示例输入：[输入内容]
示例输出：[输出内容]
```

**示例**：
```

### 示例设计原则

#### 相关性
示例应该与实际任务相关。

**相关示例**：
```
任务：审查代码质量
示例：展示代码审查的输出格式
```

#### 代表性
示例应该代表典型的输入和期望的输出。

**代表性示例**：
```
任务：情感分析
示例：展示不同情感（正面、负面、中性）的分析结果
```

#### 简洁性
示例应该简洁清晰，突出关键点。

**简洁示例**：
```
输入："这个功能很糟糕"
输出：情感：负面，强度：强，关键点：功能差
```

#### 完整性
示例应该展示完整的输出结构。

**完整示例**：
```
输入："优化数据库查询"
输出：
## 问题诊断
- 问题：查询效率低
- 原因：缺少索引

## 优化方案
1. 添加索引到字段X
2. 重构查询语句
3. 使用缓存

## 预期效果
查询时间从2秒降到0.1秒
```

```

### 常见错误

**错误1：示例质量差**
- ❌ 提供模糊、不准确的示例
- ✅ 提供高质量、清晰的示例

**错误2：示例不相关**
- ❌ 示例与实际任务不匹配
- ✅ 示例代表典型的使用场景

**错误3：示例太少**
- ❌ 只有一个示例
- ✅ 提供2-3个相关示例

**错误4：示例过多**
- ❌ 提供5+个示例，占用太多token
- ✅ 2-3个代表性示例足够

### 最佳实践
1. **质量优先**：示例质量比数量更重要
2. **多样化**：涵盖不同的输入类型
3. **标注清晰**：明确标注哪些是示例
4. **保持简洁**：每个示例聚焦关键点
5. **对齐目标**：确保示例与任务目标一致
