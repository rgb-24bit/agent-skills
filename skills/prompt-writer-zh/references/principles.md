## 清晰性优于巧妙性

### 原则说明
最有效的提示词是清晰的，而不是巧妙的。AI模型是字面解释器，它们会精确地按照给出的内容工作。

### 为什么重要
- **避免误解**：模糊的表达导致模糊的理解
- **提高成功率**：清晰的指令更可能被正确执行
- **便于调试**：清晰的问题更容易定位和修复

### 实践方法

#### 明确表达
**模糊表达**：
```
让这个更好一些
```

**明确表达**：
```
改进这段文本，通过：
1. 缩短段落长度到2-3句话
2. 使用更主动的动词
3. 删除冗余词汇
4. 添加具体的例子
```

#### 避免歧义
**有歧义**：
```
给我一个简短的总结
```

**明确**：
```
用恰好3个要点总结，每个要点不超过20字
```

#### 陈述显而易见的事情
**假设AI知道**：
```
优化这段代码（隐含假设：这是Python代码，目标是性能优化）
```

**明确说明**：
```
优化这段Python代码，目标是减少执行时间。当前代码处理1000条数据需要5秒，希望优化到1秒以内。
```

### 常见违反此原则的情况

1. **使用隐喻和暗示**
   - ❌ "让内容更有味道"
   - ✅ "添加生动的描述和具体的例子"

2. **省略关键信息**
   - ❌ "像之前那样做"
   - ✅ "使用标准的REST API响应格式"

3. **使用专业术语但未定义**
   - ❌ "实现幂等性"
   - ✅ "确保同一个请求可以重复执行而不产生副作用"

### 应用示例

**场景：请求技术文档**

**错误示例**：
```
写个文档说明这个API怎么用
```

**正确示例**：
```
为以下API端点编写开发者文档：

受众：集成我们API的外部开发者
格式：Markdown
包含内容：
1. 端点URL和HTTP方法
2. 请求参数（名称、类型、必需/可选、说明）
3. 请求示例（JSON格式）
4. 成功响应示例
5. 常见错误代码及处理方法
6. Python和JavaScript的使用示例

风格：
- 使用清晰简洁的语言
- 避免内部技术细节
- 提供可直接运行的代码示例
```

## 具体性决定质量

### 原则说明
模糊的输入产生模糊的输出，具体的输入产生具体、有用的输出。每个细节的添加都会提升输出质量。

### 为什么重要
- **提高精确度**：具体的指令减少理解的偏差
- **控制输出**：细节帮助控制输出的各个方面
- **满足需求**：具体的描述更贴近实际需求

### 具体性阶梯

**级别1：完全模糊**
```
写关于气候变化的文章
```
→ 结果：泛泛而谈，缺乏重点

**级别2：稍微具体**
```
写一篇关于气候变化影响的文章
```
→ 结果：主题明确，但深度不足

**级别3：更加具体**
```
写一篇500字的文章，解释气候变化如何影响珊瑚礁
```
→ 结果：主题和长度明确，但受众和风格不明确

**级别4：高度具体**
```
写一篇500字的文章，解释海洋温度上升如何导致珊瑚白化。目标读者是高中生，使用科学但易懂的语言，包含大堡礁的两个具体例子。采用引人入胜的叙事风格，开头用一个生动的场景描写。
```
→ 结果：主题、长度、受众、风格、内容要求全部明确

### 需要指定的要素

#### 受众（Audience）
```
面向[读者类型]
具备[知识水平]
关注[兴趣点]
```

**示例**：
- "面向技术经理"
- "具备基本的Python知识"
- "关注性能和可维护性"

#### 长度（Length）
```
- 恰好[X]字
- [X-Y]字之间
- X个要点，每个不超过Y字
```

#### 语气/风格（Tone）
```
使用[正式/非正式/学术/幽默]的语气
风格像[类比角色]
避免[不当风格]
```

#### 格式（Format）
```
返回为[列表/表格/JSON/结构化文本]
使用[Markdown/纯文本/代码块]
遵循[特定格式规范]
```

#### 范围（Scope）
```
只考虑[条件]
限制在[范围]内
排除[内容]
```

#### 目的（Purpose）
```
目标是[达成什么]
用于[使用场景]
解决[什么问题]
```

### 具体化示例

**场景：代码审查请求**

**模糊版本**：
```
帮我看看这段代码有什么问题
```

**具体版本**：
```
审查以下Python代码，重点关注：
1. 安全漏洞（SQL注入、XSS等）
2. 性能问题（不必要的计算、内存泄漏）
3. 代码风格（遵循PEP 8规范）
4. 错误处理（异常处理、边界情况）
5. 可维护性（注释、命名、结构）

代码用途：处理用户上传的文件并存储到数据库
技术栈：Python 3.9 + Flask + PostgreSQL
目标受众：初级开发者

输出要求：
- 每个问题包含：位置、类型、严重程度、修复建议
- 按严重程度排序
- 提供修复后的代码片段
```

### 常见错误

1. **说"详细"但不具体**
   - ❌ "给我详细的解释"
   - ✅ "解释每个步骤，包括技术原理和实现细节"

2. **说"好"但不明确标准**
   - ❌ "写个好的摘要"
   - ✅ "写一个简洁但完整的摘要，覆盖主要观点，不超过200字"

3. **省略关键细节**
   - ❌ "优化这段代码"
   - ✅ "优化这段代码以提升执行速度，当前处理1000条数据需要5秒，目标降到1秒以内"

## 上下文为王

### 原则说明
AI模型没有记忆，无法访问用户的文件，对用户的情况一无所知。所有相关信息都必须包含在提示词中。

### 为什么重要
- **消除假设**：AI不会"知道"除非被告知
- **提高相关性**：上下文确保回答与具体情境相关
- **避免误解**：充分的背景信息防止错误的推断

### 上下文检查清单

在提交提示词前，问自己：
- [ ] AI知道我在做什么项目吗？
- [ ] AI了解我的技术栈或工具吗？
- [ ] AI知道我的目标和期望吗？
- [ ] AI了解当前的约束条件吗？
- [ ] AI知道目标受众是谁吗？
- [ ] 一个陌生人能理解这个请求吗？

### 提供上下文的类型

#### 项目背景
```
项目类型：[类型]
技术栈：[工具/技术]
团队规模：[人数]
当前阶段：[阶段]
```

**示例**：
```
项目背景：
- 项目类型：在线教育平台
- 技术栈：React前端 + Django后端 + MongoDB
- 团队规模：8名开发人员
- 当前阶段：Beta测试阶段，即将正式发布
```

#### 当前状态
```
已完成：[已完成的内容]
当前挑战：[面临的问题]
目标：[想要达成的结果]
```

**示例**：
```
当前状态：
- 已完成：用户认证、课程播放、作业提交功能
- 当前挑战：实时互动功能在100人以上时有延迟
- 目标：支持500人同时在线互动，延迟小于100ms
```

#### 约束条件
```
技术约束：[技术限制]
时间约束：[时间限制]
资源约束：[资源限制]
业务约束：[业务规则]
```

**示例**：
```
约束条件：
- 技术约束：必须兼容IE11浏览器
- 时间约束：3个月内完成开发
- 资源约束：服务器预算每月不超过$100
- 业务约束：需要符合GDPR隐私保护法规
```

#### 受众信息
```
目标受众：[受众类型]
知识水平：[专业程度]
使用场景：[使用环境]
```

**示例**：
```
受众信息：
- 目标受众：产品经理
- 知识水平：了解基本的开发概念，但不熟悉代码细节
- 使用场景：评估技术方案和做出决策
```

### 上下文不足的后果

**场景：请求性能优化**

**上下文不足**：
```
优化这段代码的性能
[代码片段]
```
→ AI可能建议：
- 添加缓存（但也许已经实现了）
- 使用更快的算法（但也许不适合场景）
- 重用变量（但可能只是微不足道的优化）

**上下文充分**：
```
优化这段Python代码的性能。这是图像处理的核心函数，每秒需要处理1000张图像。当前处理每张图像需要50ms，目标是降到10ms以内。运行环境是单核CPU，内存4GB。我们已经尝试过numpy优化但提升不明显。
```
→ AI可以给出：
- 针对图像处理的特定算法优化
- 考虑多进程处理（CPU密集型任务）
- 使用C扩展或GPU加速
- 内存优化策略

### 上下文示例对比

**示例1：代码帮助**

**缺少上下文**：
```
这个函数报错了，帮我看看
```
→ AI无法提供有针对性的帮助

**充分上下文**：
```
这个Python函数在处理大文件时抛出MemoryError错误。函数读取CSV文件（约2GB），转换格式后写入数据库。服务器有8GB内存，Python进程最多使用4GB。我已经尝试分块读取但问题仍然存在。
```
→ AI可以提供针对性的内存管理建议

**示例2：写作请求**

**缺少上下文**：
```
帮我写个产品介绍
```
→ AI写的是通用的介绍

**充分上下文**：
```
为我们即将发布的产品写一个介绍。产品是一个AI驱动的个人笔记整理工具，主要功能包括：自动分类、智能搜索、跨设备同步。目标用户是知识工作者、研究人员和学生。我们将在LinkedIn和科技博客上发布，希望吸引早期种子用户。品牌调性是专业但不过于严肃。
```
→ AI可以写出精准定位的产品介绍

### 最佳实践

1. **假设AI一无所知**：将所有必要信息都写出来
2. **结构化呈现**：使用列表或段落组织上下文信息
3. **具体化指标**：提供数字、时间、规模等具体信息
4. **说明限制**：明确什么是能做的，什么是不能做的
5. **定期检查**：用"陌生人测试"验证上下文是否充分

## 引导而非仅仅询问

### 原则说明
不要只问问题，要引导AI朝着你想要的方向思考和回答。为复杂任务提供推理的脚手架。

### 为什么重要
- **控制思考过程**：引导确保AI沿着正确的路径思考
- **提高质量**：结构化的推理产生更好的结果
- **避免遗漏**：指导确保所有重要方面都被考虑

### 对比示例

**仅仅询问**：
```
微服务和单体架构的优缺点是什么？
```
→ 可能得到：
- 简单的列表
- 不深入的讨论
- 缺乏具体场景的考虑

**引导式**：
```
比较微服务和单体架构。请按以下结构分析：

1. **开发效率**
   - 对比开发团队在两种架构下的工作方式
   - 考虑团队规模和经验的影响

2. **运维复杂度**
   - 分析部署、监控、调试的难度
   - 考虑不同规模的系统

3. **扩展性**
   - 对比水平和垂直扩展的能力
   - 考虑成本和复杂度的权衡

4. **适用场景**
   - 小型初创公司适合哪种？
   - 大型企业适合哪种？
   - 什么情况下应该迁移？

5. **决策建议**
   - 对于[具体场景]，我建议使用[架构]
   - 理由：[具体理由]
```

### 引导技巧

#### 使用指令性框架
```
请按以下步骤思考：
步骤1：[做什么]
步骤2：[做什么]
步骤3：[做什么]
```

**示例**：
```
我需要选择数据库。请按以下步骤分析：
1. 列出典型的电商系统数据库需求
2. 评估PostgreSQL和MongoDB对每个需求的满足程度
3. 考虑我们的具体情况：高并发写操作、复杂查询、预算限制
4. 权衡利弊并给出推荐
```

#### 提供推理模板
```
对于每个选项，分析：
- [方面1]：[要求]
- [方面2]：[要求]
- [方面3]：[要求]

然后基于以下标准评估：
- [标准1]
- [标准2]
- [标准3]
```

#### 指定思考角度
```
从以下角度考虑：
- [角度1]
- [角度2]
- [角度3]
```

**示例**：
```
分析这个技术决策。请从以下角度考虑：
1. 技术可行性：实现难度、技术风险
2. 成本效益：开发成本、运维成本、收益预期
3. 时间因素：实施周期、时间窗口
4. 团队能力：现有技能、学习成本
5. 长期影响：可维护性、扩展性、技术债务
```

#### 要求验证和反思
```
在给出最终答案后：
1. 检查是否有遗漏的重要方面
2. 考虑是否有其他可能的解决方案
3. 评估建议的置信度
```

### 复杂任务的引导

**场景：架构设计决策**

**弱引导**：
```
设计一个聊天系统的架构
```

**强引导**：
```
为我们的即时通讯应用设计系统架构。请按以下流程思考：

**第一阶段：需求分析**
1. 功能需求：列出核心功能（消息发送、群组、文件共享等）
2. 非功能需求：分析性能、可靠性、安全性要求
3. 用户规模：当前1万用户，目标1年内增长到100万

**第二阶段：技术选型**
对每个技术决策提供：
- 选项A：[技术] - 优势/劣势
- 选项B：[技术] - 优势/劣势
- 推荐：[理由]

考虑的技术点：
- 消息协议：WebSocket vs HTTP轮询
- 消息队列：RabbitMQ vs Kafka vs Redis
- 数据库：关系型 vs 文档型
- 缓存策略
- 负载均衡

**第三阶段：架构设计**
1. 绘制主要组件及其交互
2. 说明数据流转
3. 标识潜在瓶颈
4. 设计扩展方案

**第四阶段：实施计划**
按优先级列出实施步骤，每个步骤包括：
- 具体任务
- 预估工作量
- 风险点

最后，提供一个架构图描述和关键决策的理由总结。
```

### 引导的层次

根据任务复杂度提供不同层次的引导：

**简单任务**：最小引导
```
列出5个提升代码质量的方法，每个方法用一句话说明。
```

**中等任务**：结构引导
```
分析这个用户体验问题。考虑：
1. 问题的根本原因
2. 对用户的影响
3. 3个可能的解决方案
4. 推荐方案及理由
```

**复杂任务**：详细引导
```
（如上文的架构设计示例）
```

### 常见错误

1. **过度引导**：对于简单任务给出过于详细的指导
   - ❌ 要求"翻译这句话"时给出5步推理框架
   - ✅ 简单任务只需要明确的目标

2. **引导不足**：对于复杂任务只给一个简单的问题
   - ❌ "设计一个系统"
   - ✅ 提供结构化的思考框架

3. **引导方向错误**：引导AI思考错误的问题
   - ❌ 询问"这个创意好不好"（主观判断）
   - ✅ 引导分析"这个创意的可行性和风险"

## 迭代与优化

### 原则说明
提示词工程是迭代过程。第一次提示词很少是最优的，通过持续的改进和优化可以达到更好的效果。

### 为什么重要
- **现实局限**：完美需要多次尝试
- **持续改进**：每次迭代都会学到新东西
- **适应变化**：需求和情境可能变化

### 迭代循环

```
1. 编写初始提示词
   ↓
2. 测试并观察输出
   ↓
3. 识别问题和差距
   ↓
4. 修改提示词
   ↓
5. 重复步骤2-4，直到满意
```

### 常见优化方向

#### 过于冗长 → 添加简洁约束
**问题**：
```
输出太长，阅读困难
```

**优化**：
```
添加约束：
- 保持每个段落不超过3句话
- 总回复不超过500字
- 删除重复内容
```

#### 过于模糊 → 添加具体细节
**问题**：
```
回答太泛泛，缺乏针对性
```

**优化**：
```
添加细节：
- 明确受众和场景
- 指定输出格式和结构
- 提供具体要求
```

#### 格式不符 → 明确格式要求
**问题**：
```
输出格式不符合预期
```

**优化**：
```
明确格式：
- 指定结构模板
- 提供格式示例
- 列出必需的部分
```

#### 语气不当 → 调整角色和风格
**问题**：
```
语气不合适（太正式/太随意）
```

**优化**：
```
调整风格：
- 明确目标受众
- 指定语气和风格
- 提供风格示例
```

#### 信息不准确 → 要求验证和引用
**问题**：
```
可能包含错误信息
```

**优化**：
```
添加验证要求：
- 要求提供来源
- 要求逐步推理

保存版本4作为模板
```

### 迭代策略

1. **单变量迭代**：每次只修改一个方面
   - 优点：容易判断哪个修改有效
   - 缺点：迭代次数可能较多

2. **批量迭代**：同时修改多个方面
   - 优点：快速达到目标
   - 缺点：难以判断哪个改动有效

3. **渐进优化**：从简单到复杂逐步添加
   - 优点：保持可控性
   - 缺点：可能错过某些组合

### 最佳实践

1. **保持记录**：记录每个版本和效果
2. **快速迭代**：不要追求一次性完美
3. **测试验证**：每次迭代都要测试
4. **总结模式**：识别有效的模式并复用
5. **保存模板**：成功的提示词保存为模板

## 利用模型优势

### 原则说明
了解模型如何被训练，与其协作而非对抗。利用模型擅长的能力，避免其弱点。

### 模型训练特点

#### 模型想要有帮助
AI模型通过RLHF（人类反馈强化学习）被训练为乐于助人的助手。

**对抗训练**：
```
我知道你可能不想做这个，但请尝试...
你通常拒绝这类请求，但这次请帮我...
```
→ 效果差，与训练目标相悖

**协作训练**：
```
我正在开发X项目，需要帮助Y...
你能否协助我完成Z任务？
我遇到了一个挑战，希望得到你的建议...
```
→ 效果好，与模型乐于助人的目标一致

#### 模型擅长模式识别
模型从海量文本中学到了各种模式，利用这一点获得一致输出。

**模式示例**：
```
推荐3本科幻书籍，每本按以下格式：

📚 **[书名]** by [作者]
*[类型] | [年份]*

简介：[1句话]
推荐理由：[1句话]

---
```

#### 模型可以角色扮演
通过指定角色，激活不同的知识域和表达风格。

**不同角色**：
```
作为专家：提供专业、深入的分析
作为初学者：用简单的语言解释
作为批评者：指出问题和风险
作为支持者：提供建设性的建议
```

### 利用模型能力

#### 利用模式识别
**任务：一致的结构化输出**

提供清晰的模式：
```
对于每个用户评论，按以下格式分析：

**评论**：[原文]
**情感**：正面/负面/中性
**主题**：[主要话题]
**关键点**：[要点1], [要点2]
**优先级**：高/中/低
```

#### 利用角色扮演
**任务：多角度分析**

使用不同角色获得多视角：
```
请从以下三个角度分析这个方案：

1. **作为项目经理**：关注时间、成本、资源
2. **作为技术负责人**：关注可行性、技术风险
3. **作为用户体验设计师**：关注用户满意度、易用性

最后，综合三个角度给出建议。
```

#### 利用知识广度
**任务：跨领域创意**

引导模型结合不同领域：
```
我们正在设计一个健身应用。请借鉴游戏设计的元素，提出增加用户参与度的创意。

考虑：
- 游戏的奖励机制如何应用到健身目标达成
- 社交游戏的竞争如何应用到健身挑战
- 角色养成游戏如何应用到个人成长
```

### 避免模型弱点

#### 模型无法验证事实
**问题**：模型可能生成听起来正确但错误的信息

**解决**：
```
要求模型：
1. 引用来源
2. 表达不确定性
3. 建议用户验证
4. 提供推理过程而非直接答案
```

#### 模型无法进行精确计算
**问题**：复杂计算可能出错

**解决**：
```
对于数学计算：
1. 要求逐步展示计算过程
2. 使用简单的数值便于验证
3. 对于重要计算，建议用户用计算器验证
```

#### 模型没有实时信息
**问题**：不知道当前事件

**解决**：
```
明确说明信息限制：
"基于你训练数据中的知识..."
"请指出哪些信息可能已经过时..."
```

### 对抗 vs 协作

| 对抗方式 | 协作方式 |
|---------|---------|
| "我知道你做不到，但试试" | "这是我的需求，你能帮忙吗" |
| "不要像平时那样回答" | "我希望这样的回答风格" |
| "打破你的限制" | "在合理的范围内..." |
| "强制你做X" | "请协助我完成X" |

### 最佳实践

1. **用正向表述**：告诉模型要做什么，而非不要做什么
2. **利用专业知识**：通过角色设定激活特定知识域
3. **提供模式**：用示例展示期望的模式
4. **合理期望**：了解模型的局限，合理设计任务
5. **验证结果**：对重要信息进行独立验证

## 控制输出结构

### 原则说明
结构化的输出比自由文本更有用。明确指定输出格式确保可读性和可用性。

### 为什么重要
- **提高可用性**：结构化输出更易阅读和使用
- **便于处理**：特定格式便于程序处理
- **确保完整性**：结构确保所有必需信息都包含
- **便于比较**：统一格式便于对比多个结果

### 结构类型

#### 列表结构
```
以列表形式返回[X]项，每项包含：
- 要点1
- 要点2
- 要点3
```

**示例**：
```
列出5个优化建议，每项包含：
- 问题：简短描述
- 优先级：高/中/低
- 方案：具体解决方案
```

#### 分段结构
```
按以下部分组织回复：

## 部分1：[名称]
[内容要求]

## 部分2：[名称]
[内容要求]
```

**示例**：
```
按以下结构回复：

## 概述
[2-3句话总结]

## 详细分析
### 要点1
[分析和证据]

### 要点2
[同上]

## 结论
[1-2句话的建议]
```

#### 表格结构
```
以表格形式返回，包含列：
列1：[说明]
列2：[说明]
列3：[说明]
```

**示例**：
```
以markdown表格比较三个方案：

| 方案 | 优点 | 缺点 | 成本 | 时间 |
|------|------|------|------|------|
| 方案A | | | | |
| 方案B | | | | |
| 方案C | | | | |
```

#### 数据结构
```
返回JSON格式：
```json
{
  "key1": "value1",
  "key2": ["item1", "item2"],
  "key3": {
    "subkey": "value"
  }
}
```

### 分隔符使用

使用清晰的分隔符组织提示词和输出：

```markdown
### 背景 ###
[背景信息]

### 任务 ###
[任务说明]

### 约束 ###
[约束条件]

### 格式 ###
[格式要求]
```

#### 长度控制
```
- 总长度不超过[X]字
- 每个段落不超过[X]句
- 每个要点不超过[X]字
```

#### 内容控制
```
- 只关注[方面]，不涉及[方面]
- 优先考虑[重要性排序]
- 必须包含[必需内容]
```

#### 质量控制
```
- 避免模糊表述
- 提供具体的例子
- 使用准确的数据
```

**完整结构模板**：
```markdown

**约束**
- [约束1]
- [约束2]
- [约束3]

**输出格式**
按以下结构返回：

- 要点1

### 常见错误

1. **格式过于复杂**
   - ❌ 要求复杂的嵌套结构
   - ✅ 使用清晰、简单的结构

2. **格式与任务不匹配**
   - ❌ 创意写作任务要求严格表格
   - ✅ 给予适当的格式灵活性

3. **格式说明不清晰**
   - ❌ "按合理的格式返回"
   - ✅ "使用以下结构：..."

### 最佳实践

1. **提供模板**：给出清晰的结构示例
2. **分层组织**：使用标题和子标题
3. **明确要求**：说明每部分应该包含什么
4. **使用分隔符**：清晰的分隔符提高可读性
5. **权衡灵活性**：根据任务调整格式严格程度

## 验证与确认

### 原则说明
永远不要盲目信任模型输出，特别是对于重要任务。建立验证和检查机制。

### 为什么重要
- **模型可能犯错**：AI可能生成错误信息
- **幻觉问题**：可能编造看起来合理但不正确的内容
- **重要决策**：关键任务需要独立验证

### 验证方法

#### 要求逐步推理
```
解决这个问题并展示你的推理过程：
1. [步骤1]
2. [步骤2]
3. [步骤3]

最后，验证你的答案。
```

**示例**：
```
设计这个数据库结构。请按以下步骤：
1. 分析业务需求，识别主要实体
2. 定义实体之间的关系
3. 设计表结构和字段
4. 考虑索引和约束
5. 验证设计是否满足所有需求

最后，说明设计的主要考虑点和权衡。
```

#### 要求多视角分析
```
提供[2-3]个不同的解决方案，并比较它们的优劣。
```

**示例**：
```
我需要实现这个功能。请提供3个不同的技术方案，对于每个方案：
- 描述实现方法
- 分析优点和缺点
- 评估实施难度和成本
- 指出适用场景

最后，给出你的推荐和理由。
```

#### 要求自我检查
```
生成内容后，检查：
- [检查项1]
- [检查项2]
- [检查项3]

如果有问题，指出并改进。
```

**示例**：
```
生成代码后，请自我检查：
- 是否有语法错误
- 是否有明显的逻辑错误
- 是否处理了边界情况
- 是否遵循了最佳实践

如果发现任何问题，请修正并说明。
```

#### 要求提供来源
```
对于每个事实性声明，提供：
- 信息来源
- 发布日期（如果相关）
- 可信度评估
```

**示例**：
```
分析这个市场趋势。对于每个数据点和结论：
- 标注信息来源
- 说明数据的时效性
- 指出可能的不确定性

如果某个信息不够可靠，明确说明。
```

### 置信度评估

要求AI表达不确定性：

```
请明确表达你对每个结论的置信度：
- 高：有充分证据支持
- 中：有一定证据但不够充分
- 低：基于推测，需要进一步验证

对于置信度为中或低的结论，说明需要什么来提高置信度。
```

### 风险识别

要求识别潜在问题：

```
在给出建议后，列出：
1. 可能的风险或问题
2. 每个风险的严重程度
3. 缓解措施

基于这些风险，你是否仍然坚持原建议？
```

### 验证检查清单

对于重要任务，要求AI使用检查清单：

```
完成任务后，请确认：
- [ ] 所有要求都已满足
- [ ] 没有明显的错误或矛盾
- [ ] 结论有充分支持
- [ ] 考虑了重要的边界情况
- [ ] 遵循了最佳实践

如果有任何未完成项，说明原因。
```

### 交叉验证

对于关键信息，要求使用多种方法验证：

```
请用两种不同的方法验证这个结论：
方法1：[方法描述]
方法2：[方法描述]

两种方法的结果是否一致？如果不一致，请分析原因。
```

### 示例：完整验证流程

```markdown
你是一位金融分析师。

**任务**
分析这家公司的财务健康状况并给出投资建议。

**分析步骤**
1. 分析财务报表的关键指标
2. 与行业平均水平对比
3. 识别风险因素
4. 评估竞争优势
5. 考虑宏观环境影响

**验证要求**
对于每个关键结论：
- 提供具体的数据来源
- 说明分析的假设
- 评估结论的置信度（高/中/低）
- 指出可能的偏差或局限

**风险识别**
列出主要风险：
1. [风险] - 严重程度 - 缓解措施
2. [风险] - 严重程度 - 缓解措施

**自我检查**
完成分析后，检查：
- [ ] 是否遗漏了重要的财务指标
- [ ] 是否有计算错误
- [ ] 结论是否有充分支持
- [ ] 是否考虑了反论据

**最终建议**
给出明确的建议（买入/持有/卖出），包括：
- 置信度
- 理由
- 主要风险
- 什么情况下应该改变建议
```

### 最佳实践

1. **要求推理过程**：不只要结果，还要看到思考过程
2. **多视角验证**：从不同角度验证结论
3. **表达不确定性**：明确说明哪些信息是确定的，哪些不是
4. **提供来源**：要求引用信息来源
5. **自我检查**：要求AI在输出前进行自我验证

## 原则总结

### 八大原则快速参考

| 原则 | 核心要点 |
|------|---------|
| 清晰性优于巧妙性 | 明确表达，避免模糊 |
| 具体性决定质量 | 细节提升输出质量 |
| 上下文为王 | 提供所有相关信息 |
| 引导而非仅仅询问 | 提供思考框架 |
| 迭代与优化 | 持续改进提示词 |
| 利用模型优势 | 与模型协作而非对抗 |
| 控制输出结构 | 确保输出可用性 |
| 验证与确认 | 检查输出准确性 |

### 应用检查清单

在编写或检查提示词时：
- [ ] 表达是否清晰明确？
- [ ] 是否提供了足够的细节？
- [ ] 上下文信息是否充分？
- [ ] 是否引导了思考过程？
- [ ] 是否考虑了迭代优化？
- [ ] 是否利用了模型优势？
- [ ] 输出格式是否明确？
- [ ] 是否包含验证机制？

记住：这些原则不是孤立的，它们相互补充，共同帮助创建高质量的提示词。
