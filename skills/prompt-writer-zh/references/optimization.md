## 常见问题诊断

### 问题1：输出过于泛泛

**症状**：
- 回答缺乏针对性
- 内容不够深入
- 没有解决具体问题

**可能原因**：
- 缺少角色设定
- 上下文不充分
- 任务不够具体
- 没有明确受众

**诊断检查**：
- [ ] 提示词中是否指定了角色？
- [ ] 是否提供了足够的背景信息？
- [ ] 任务描述是否具体？
- [ ] 是否说明了目标受众？

**优化方向**：
1. 添加角色定义
2. 补充上下文信息
3. 明确任务的具体要求
4. 指定输出深度和重点

---

### 问题2：输出过长或过短

**症状**：
- 输出内容冗长，包含大量无关信息
- 输出过于简短，缺少关键信息

**可能原因**：
- 缺少长度约束
- 任务范围不明确
- 没有指定重点

**诊断检查**：
- [ ] 是否设置了长度约束？
- [ ] 任务范围是否明确？
- [ ] 是否指定了重点关注的方面？

**优化方向**：
1. 添加明确的长度限制
2. 明确任务的边界
3. 指定重点和排除的内容

---

### 问题3：格式不符合预期

**症状**：
- 输出格式混乱
- 结构不清晰
- 难以阅读或使用

**可能原因**：
- 没有指定输出格式
- 格式要求不清晰
- 缺少结构模板

**诊断检查**：
- [ ] 是否指定了输出格式？
- [ ] 格式要求是否清晰？
- [ ] 是否提供了结构模板？

**优化方向**：
1. 明确指定输出格式（列表、表格、JSON等）
2. 提供清晰的结构模板
3. 给出格式示例

---

### 问题4：信息不准确或幻觉

**症状**：
- 包含错误信息
- 编造不存在的事实
- 引用不存在的来源

**可能原因**：
- 要求模型处理超出其知识范围的内容
- 没有要求验证
- 缺少推理过程要求

**诊断检查**：
- [ ] 任务是否在模型能力范围内？
- [ ] 是否要求模型验证信息？
- [ ] 是否要求提供推理过程？

**优化方向**：
1. 要求提供信息来源
2. 要求逐步推理
3. 要求表达不确定性
4. 建议独立验证

---

### 问题5：缺少必要组件

**症状**：
- 输出不完整
- 缺少关键部分
- 结构不完整

**可能原因**：
- 提示词缺少某些核心组件
- 输出格式要求不完整

**诊断检查**：
使用组件检查清单：
- [ ] 是否有角色设定？
- [ ] 是否有上下文信息？
- [ ] 是否有明确的任务？
- [ ] 是否有约束条件？
- [ ] 是否有输出格式要求？
- [ ] 是否有示例？

**优化方向**：
补充缺失的组件

---

### 问题6：重复或冗余内容

**症状**：
- 内容重复
- 表达冗余
- 信息重复出现

**可能原因**：
- 任务表述重复
- 约束条件重叠
- 没有要求精简

**诊断检查**：
- [ ] 提示词中是否有重复的表述？
- [ ] 是否添加了简洁性约束？

**优化方向**：
1. 清理提示词中的重复表述
2. 添加简洁性要求
3. 明确要求避免冗余

---

### 问题7：风格或语气不当

**症状**：
- 语气过于正式或过于随意
- 风格不符合预期
- 用词不当

**可能原因**：
- 没有指定角色的人设
- 没有说明目标受众
- 缺少风格约束

**诊断检查**：
- [ ] 是否指定了角色的沟通风格？
- [ ] 是否说明了目标受众？
- [ ] 是否有风格约束？

**优化方向**：
1. 明确指定沟通风格
2. 说明目标受众
3. 添加风格示例

## 优化策略

### 策略1：添加角色设定

**适用场景**：输出缺乏专业深度或特定视角

**方法**：
```
你是一位[职业]，具有[年限]年经验，专长于[领域]。
你的特点是[风格特征]。
```

**示例**：
```
优化前：
审查这段代码

优化后：
你是一位具有10年经验的资深Python后端工程师，专长于高并发系统设计和性能优化。你善于发现隐藏的性能瓶颈，并提供实用的优化建议。

审查这段代码：
[代码]
```

---

### 策略2：补充上下文信息

**适用场景**：输出不相关或不准确

**方法**：
提供项目背景、当前状态、约束条件、受众信息

**示例**：
```
优化前：
帮我优化这个查询

优化后：
帮我优化这个SQL查询。

背景信息：
- 数据库：PostgreSQL 13
- 表规模：users表有100万行
- 查询频率：每分钟约100次
- 当前性能：平均响应时间2秒
- 目标性能：响应时间降到0.5秒以内
- 索引情况：已在user_id字段上建立索引

查询语句：
[SQL]
```

---

### 策略3：明确任务和输出格式

**适用场景**：输出结构混乱或不完整

**方法**：
使用具体动词，明确任务范围，指定输出结构

**示例**：
```
优化前：
分析这个产品

优化后：
分析这款智能手表的用户体验。

任务：
1. 识别3个主要的用户体验优点
2. 识别3个主要的用户体验问题
3. 为每个问题提供具体的改进建议
4. 评估改进的可行性和成本

输出格式：
## 优点分析
- 优点1：[名称] - [说明]

## 问题分析
- 问题1：[名称] - [影响] - [严重程度]

## 改进建议
- 建议1：[方案] - [实施难度] - [预期效果]
```

---

### 策略4：添加约束条件

**适用场景**：输出过长、过于复杂或不符合要求

**方法**：
设置长度、内容、风格、范围约束

**示例**：
```
优化前：
解释机器学习

优化后：
为高中生解释机器学习的基本概念。

约束条件：
- 使用类比和实例，避免复杂的数学公式
- 总长度不超过500字
- 分为3个段落：概念解释、应用场景、学习建议
- 语言生动有趣，避免过于学术化
- 重点解释"什么是机器学习"而非技术细节
```

---

### 策略5：提供示例

**适用场景**：输出格式或质量不达标

**方法**：
提供1-3个输入-输出示例

**示例**：
```
优化前：
总结这些评论

优化后：
总结用户评论，提取主要情感和关键问题。

示例：
输入："这个APP经常崩溃，用户体验很差"
输出：情感：负面，关键问题：稳定性

输入："功能很强大，但是操作太复杂了"
输出：情感：正面但有待改进，关键问题：易用性

现在总结以下评论：
[评论列表]
```

---

### 策略6：使用框架重构

**适用场景**：提示词结构混乱或缺少关键组件

**方法**：
使用CRISPE、RTF或RISE框架重新组织

**示例**：
```
优化前（混乱的提示词）：
我是做电商的，帮我写个推广文案，要吸引人，别太长，我们卖的是护肤品，目标用户是25-35岁的女性

优化后（使用CRISPE框架）：
**Capacity**：你是一位具有5年经验的资深营销文案撰写人，专长于美妆护肤品的品牌营销。

**Request**：为我们的新护肤产品撰写一份推广文案。

**Information**：
- 产品：天然有机护肤品系列
- 目标用户：25-35岁的城市女性，注重生活品质和天然成分
- 品牌定位：专业、可信、温和有效
- 核心卖点：纯天然成分、无添加、临床验证有效

**Situation**：新产品即将在本月上市，将在社交媒体和小红书平台推广。这是首次向目标用户介绍产品，需要建立品牌认知。

**Persona**：使用亲切、专业的语气，像一位懂护肤的朋友在真诚推荐。避免过度营销的语言，以分享真实体验的方式表达。

**Experiment**：
示例风格：
"作为用了半年的用户真心推荐，这款精华液真的改善了我的敏感肌..."

**输出要求**：
- 长度：300-400字
- 格式：适合社交媒体发布
- 内容：包含产品亮点、使用感受、适用人群
```

---

### 策略7：添加验证要求

**适用场景**：需要确保输出准确性

**方法**：
要求推理过程、多视角分析、自我检查

**示例**：
```
优化前：
设计这个数据库

优化后：
设计这个电商网站的数据库结构。

要求：
1. 按以下步骤分析：
   - 识别业务实体
   - 定义实体关系
   - 设计表结构
   - 考虑索引和约束

2. 自我检查：
   - 是否满足所有业务需求
   - 是否有冗余数据
   - 是否考虑了扩展性
   - 是否遵循了数据库设计原则

3. 多视角验证：
   - 从性能角度：查询效率如何
   - 从维护角度：是否易于维护
   - 从扩展角度：未来扩展是否方便

4. 标注不确定性：
   - 对于基于假设的设计，说明假设条件
   - 指出需要进一步确认的信息
```

---

### 策略8：调整复杂度和重点

**适用场景**：输出过于复杂或过于简单

**方法**：
根据任务需求调整输出的复杂度和重点

**示例**：
```
优化前（对初学者）：
解释深度学习

优化后（为初学者）：
用简单的语言解释深度学习的基本概念。假设读者了解基本的计算机概念，但不了解AI技术。

重点：
1. 用生活中的类比解释概念
2. 避免数学公式和复杂术语
3. 强调深度学习能做什么，而非如何实现
4. 提供1-2个简单易懂的应用例子

约束：
- 使用段落式文本，不用列表
- 每个段落3-4句话
- 总长度不超过300字
- 使用生动有趣的语言
```

## 优化案例

### 案例1：代码审查提示词

**版本1（初始版本）**：
```
审查这段代码
[代码]
```

**问题诊断**：
- ❌ 没有角色设定
- ❌ 没有上下文
- ❌ 审查重点不明确
- ❌ 没有输出格式要求

**版本2（添加角色和上下文）**：
```
你是一位Python开发工程师。这段代码处理用户数据，需要审查。
[代码]
```

**问题**：
- ✅ 有角色设定
- ❌ 上下文仍然不足
- ❌ 审查重点模糊
- ❌ 没有输出格式

**版本3（完善组件）**：
```
你是一位具有8年经验的资深Python开发工程师，专长于代码审查和安全审计。

上下文：
- 这是一个用户认证模块
- 使用Flask框架
- 目标用户：初级开发者

任务：
审查以下代码，重点关注：
1. 安全漏洞
2. 代码风格（PEP 8）
3. 性能问题
4. 错误处理

约束：
- 每个问题提供具体的行号
- 给出修复建议
- 保持建议对初学者友好

[代码]
```

**问题**：
- ✅ 所有组件基本完整
- ❌ 输出格式可以更结构化
- ❌ 可以添加示例

**版本4（最终优化）**：
```
你是一位具有8年经验的资深Python开发工程师，专长于代码审查和安全审计。你善于发现潜在问题并提供清晰、可操作的改进建议。

## 上下文
这是我们的用户认证模块代码，使用Flask框架开发。目标读者是初级开发者，因此建议应该清晰易懂，避免过于复杂的优化方案。

## 任务
审查以下代码，重点关注：
1. 安全漏洞（SQL注入、XSS、CSRF等）
2. 代码风格（是否符合PEP 8规范）
3. 性能问题（不必要的计算、资源泄漏）
4. 错误处理（异常处理、边界情况）

## 约束
- 每个问题提供具体的行号
- 给出可直接实施的修复建议
- 使用对初学者友好的语言
- 按严重程度排序（高→中→低）

## 输出格式
请按以下结构返回：

## 安全问题
- **问题1**：[名称]
  - 位置：行[X-Y]
  - 严重程度：高/中/低
  - 描述：[简短说明]
  - 修复：[具体代码或步骤]
  - 为什么重要：[1句话解释风险]

## 代码风格问题
[同上结构]

## 性能问题
[同上结构]

## 错误处理问题
[同上结构]

## 总体评价
- 代码质量：[评分/10]
- 主要优点：[2-3个]
- 优先修复：[列出最关键的1-2个问题]

## 示例输出格式
**问题**：SQL注入风险
- 位置：行45
- 严重程度：高
- 描述：直接拼接SQL查询字符串，未验证用户输入
- 修复：使用参数化查询
  ```python
  cursor.execute("SELECT * FROM users WHERE id = %s", (user_id,))
  ```
- 为什么重要：攻击者可以通过构造恶意输入获取或修改数据库数据

[代码片段]
```

- ❌ 角色不够具体
- ❌ 没有受众和角度

- 目标媒体：面向企业领导者和HR的专业杂志

Format:
## 标题
[吸引人但专业的标题，包含核心关键词]

## 导语（100字）
[引出话题，说明重要性，概括文章要点]

## 现状分析（300字）
### 数据支持
引用2-3个权威研究的数据
### 趋势观察
描述当前远程工作的发展趋势

- 积极面：灵活性、工作生活平衡
- 消极面：孤独感、协作挑战

- 绩效管理方式的转变
- 企业文化的维护

3条可操作的建议

[总结核心观点，展望未来，给出行动号召]

- 避免过度乐观或悲观的极端观点，保持平衡

Example Style:

```

### 案例3：问题诊断提示词

**版本1（初始版本）**：
```
我的系统很慢，帮我看看问题
```

**问题诊断**：
- ❌ 没有任何具体信息
- ❌ 不可能给出有针对性的建议

**版本2（添加上下文）**：
```
我的网站很慢。这是一个React前端和Node.js后端的电商网站，用户在浏览商品页面时加载时间超过5秒。帮我找出问题。
```

**问题**：
- ✅ 有基本上下文
- ❌ 信息仍然不够详细
- ❌ 没有系统化的分析方法

**版本3（使用RISE框架）**：
```
Role: 你是一位全栈性能优化专家，精通前端和后端性能分析，有10年大型Web应用优化经验。

## 迭代方法

### 迭代流程

```
1. 编写初始提示词
   ↓
2. 测试并记录输出
   ↓
3. 分析输出与期望的差距
   ↓
4. 识别需要改进的方面
   ↓
5. 应用优化策略
   ↓
6. 重新测试
   ↓
7. 重复步骤3-6直到满意
```

### 单变量 vs 多变量迭代

**单变量迭代**（推荐用于学习）：
- 每次只修改一个方面
- 优点：容易判断哪个修改有效
- 缺点：迭代次数较多

**示例**：
```
版本1 → 版本2：添加角色设定
版本2 → 版本3：添加上下文
版本3 → 版本4：添加格式要求
版本4 → 版本5：添加约束条件
```

**多变量迭代**（推荐有经验时）：
- 同时修改多个方面
- 优点：快速达到目标
- 缺点：难以判断哪个改动有效

### 快速迭代技巧

1. **从最小可用提示词开始**
   - 先写最基本的版本
   - 快速测试发现问题

2. **优先解决最明显的问题**
   - 先解决格式、结构等明显问题
   - 再优化内容质量

3. **使用A/B测试**
   - 创建两个不同版本的提示词
   - 比较哪个版本效果更好

4. **记录有效模式**
   - 发现有效的表述方式
   - 保存为可复用的模板

### 迭代决策树

```
输出不符合预期
    ↓
是格式问题吗？
    ├─ 是 → 调整输出格式要求
    └─ 否 → 是内容问题吗？
              ├─ 是 → 内容泛泛？→ 添加角色和具体要求
              │      内容过短？→ 调整长度约束
              │      内容过长？→ 添加简洁性约束
              │      内容不准确？→ 添加验证要求
              └─ 否 → 是结构问题吗？
                        ├─ 是 → 使用框架重构
                        └─ 否 → 是上下文问题？
                                  ├─ 是 → 补充上下文信息
                                  └─ 否 → 检查所有组件
```

### 最佳实践总结

1. **保持耐心**：好的提示词需要多次迭代
2. **记录过程**：记录每次迭代的变化和效果
3. **测试验证**：每次修改后都要测试
4. **总结模式**：识别和复用有效的模式
5. **持续学习**：从每个迭代中学习经验

记住：提示词工程是迭代的艺术，没有一蹴而就的完美提示词。通过持续的测试、分析和优化，你将逐步掌握创建高质量提示词的技能。
